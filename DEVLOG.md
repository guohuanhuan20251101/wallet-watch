# 开发日志

## 2026-06-15 · 项目启动日

### 项目起源

用户正在自学数据开发，需要一个**真实的、自己能每天用上的 GitHub 项目**放到简历里。讨论过多个方案：

| 方案 | 结论 |
|---|---|
| 数据岗位求职市场分析 | 招聘平台反爬严重，数据获取不稳定 |
| 数据质量监控引擎 (DQE) | 用户没有生产数仓，感知不到价值 |
| **个人财务分析工具** ← 选定 | 自己每月都用，数据源合法，技术栈覆盖全面 |

### 核心设计决策

**1. 星型模型而非大宽表**
- 1 张事实表 `fact_transaction` + 3 张维度表 (`dim_date`, `dim_category`, `dim_source`)
- 面试时能讲数仓建模思路，比堆一张宽表有区分度

**2. 多源数据合并按日期而非简单拼接**
- 微信和支付宝同一天的消费交错排列，而不是先列完微信再列支付宝
- 用 `sort_values(["date", "source"])` 实现

**3. 自动分类用关键字规则而非 ML**
- 13 类消费 + 关键词映射表
- 收入单独识别
- 原因：规则够用、调试方便、不依赖训练数据

**4. 消费洞察「纯数据陈述，不说教」**
```
✅ "本月外卖 23 单共 ¥856，占餐饮的 68%"
❌ "建议减少外卖，多做饭"
```
所有洞察基于数据计算，不输出任何「你应该怎样」的建议。

**5. 重叠检测交由用户决定**
- 上传新账单时检测与已有数据的日期重叠
- 用户选择：跳过重叠（保留旧数据）或覆盖

### 经验教训

1. **入口文件放根目录**：`main.py` 在 `app/` 内会导致 `from app.xxx` 找不到模块
2. **Streamlit 的 plotly_chart 需要唯一 key**：多个页面复用同一图表函数会导致 `DuplicateElementId` 错误
3. **openpyxl 不是 pandas 的默认依赖**：处理 Excel 需要单独安装
4. **沙箱推 GitHub 不稳定**：需要本地 `git push` 或换 token
